Unit 01 · Architecture

SAP Fiori Architecture Overview

A user's request travels from the browser through 7 distinct layers to reach the SAP HANA database and back. Understanding this flow is the foundation of every SAP Fiori application.

The 7-Layer Stack (click any layer to explore)
🌐
LAYER 1
End User (Browser)
Chrome · Edge · Mobile · Tablet
▾

The entry point. Users access Fiori via a browser at a URL like https://mycompany.launchpad.cfapps.... They see tiles, apps, search, and notifications — all rendered by the browser through HTTPS.

↕ HTTPS
🏠
LAYER 2
SAP Fiori Launchpad (FLP)
Tiles · Navigation · Personalization
▾

Think of FLP as the desktop of SAP. It is the unified entry point for all Fiori applications — providing tiles, groups, search, notifications, and spaces. Clicking a tile launches the corresponding SAPUI5 app.

↕ Auth Flow
🔐
LAYER 3
Authentication & Authorization
IAS · XSUAA · PFCG
▾

Authentication (IAS) — "Who are you?" — verifies identity via IAS or a corporate IdP (Azure AD).
Authorization (XSUAA / PFCG) — "What are you allowed to do?" — controls role-based access to apps and data.

↕ App Load
⚛️
LAYER 4
SAPUI5 Application (Frontend)
XML Views · Controllers · Models · Routing
▾

The entire frontend runs in the browser as JavaScript. Project structure: webapp/ containing Component.js, manifest.json, views, controllers, models, i18n, and css. SAPUI5 renders the UI and communicates with the backend via OData.

↕ OData HTTP
🔗
LAYER 5
OData Services (API Layer)
OData V2 · OData V4 · CAP · RAP
▾

The frontend never accesses the database directly. All communication goes through OData REST APIs. Methods: GET (read), POST (create), PUT/PATCH (update), DELETE. OData services expose entity sets as standard HTTP endpoints.

↕ Service Call
⚙️
LAYER 6
Business Logic
ABAP · CAP (Node.js/Java) · RAP
▾

This is where business rules are enforced — validations, calculations, workflows. Example: "Leave Days > Available Balance → Reject Request." Implemented in ABAP, CAP (Node.js or Java), or RAP behavior definitions.

↕ SQL / HANA API
🗄️
LAYER 7
Database — SAP HANA
In-memory · Column store · Real-time analytics
▾

SAP HANA is an in-memory columnar database that stores all business data — employees, sales orders, vendors, materials, purchase orders. Its in-memory architecture provides extremely fast read and analytical performance.

Layer 01 · Presentation

End User — Presentation Layer

Where the user interacts with SAP through a standard web browser. No software installation required.

Supported Devices

📱
Mobile Phone
Cozy density mode (larger touch targets). Responsive layout via sap.m controls. Optimized for manager approvals.
📟
Tablet
Mid-size form factor. SplitApp pattern shows list and detail simultaneously. Ideal for field work.
🌐

Chrome, Edge, Firefox, Safari. SAPUI5 is tested and certified across all major browsers.
What the User Sees
URL: https://mycompany.launchpad.cfapps.eu10.hana.ondemand.com User sees: ┌─────────────────────────────────────────┐ │ 🏠 SAP Fiori Launchpad │ │ │ │ [Leave Request] [My Payslip] │ │ [Travel Expense] [Timesheet] │ │ [Approve Orders] [Analytics] │ │ │ │ 🔔 Notifications 🔍 Search │ └─────────────────────────────────────────┘

Key Point: The user never installs SAP software. Everything runs in the browser as a web application. SAPUI5 is a JavaScript framework — the app is downloaded and executed client-side.
Q: Why doesn't SAP Fiori require software installation on the client device?
✓ SAP Fiori apps are built on SAPUI5, a JavaScript framework that runs entirely in the browser. The application code is served from SAP BTP or an ABAP server and executed client-side — making it a true web application with zero installation overhead.
Layer 02 · Entry Point

SAP Fiori Launchpad (FLP)

The unified shell — the "desktop of SAP." Every Fiori application is accessed through the FLP via tiles.

What FLP Provides
🔲
Application Tiles
Each tile represents one Fiori app. It shows a count, status, or KPI from the backend. Clicking launches the app.
🗂️
Groups & Spaces
Tiles are organized into groups and spaces. Admins define default layouts; users personalize their home page.
🔍
Global Search
Enterprise search across all apps and data. Powered by SAP Enterprise Search or ABAP search providers.
🔔
Notifications
Real-time alerts for workflow tasks, approvals, and alerts. Users act directly from the notification.
👤
User Menu
Personalization, language, content density (Cozy/Compact), and settings all accessible from the user menu.
📐
Content Density
FLP reads the app's manifest.json to set Compact (desktop) or Cozy (touch) density on the <body> tag automatically.
Example FLP Tile Set
Employee Self Service Space ┌──────────────────────────────────────────────────┐ │ │ │ [📅 Leave Request] [💰 My Payslip] │ │ Pending: 2 March 2026 │ │ │ │ [✈️ Travel Expense] [⏱ Timesheet] │ │ Submit by Friday 8h / 8h today │ │ │ │ [📊 My Analytics] [✅ Approve Orders] │ │ View Reports Pending: 5 │ │ │ └──────────────────────────────────────────────────┘ ↑ Clicking any tile launches the SAPUI5 app
How are tiles assigned to users? ▾
Tiles are mapped to Fiori Catalog entries and assigned via Business Roles (PFCG/XSUAA). An admin creates a role, assigns catalogs and groups, and assigns the role to users. Only tiles the user is authorized for appear on their launchpad.
What is the difference between a Catalog and a Group? ▾
A Catalog is a list of apps a user is authorized to use. A Group (or Space/Page) defines how those apps are visually arranged on the home screen. A user can have multiple catalogs and can customize how groups appear.
Q: What is the Fiori Launchpad's role in the architecture?
✓ The FLP is the unified entry point (shell) for all Fiori applications. It handles tile display, navigation, global search, notifications, and personalization. It also sets content density based on the app's manifest.json and the user's device.
Layer 03 · Security

Authentication & Authorization

Two distinct security checks happen before any app data is accessed: verifying who you are, and determining what you can do.

Authentication — "Who Are You?" (IAS)
🔑
IAS — Identity Authentication Service
SAP's cloud-based identity provider. Handles login screens, MFA (multi-factor auth), and SSO across all SAP cloud apps.
🏢
Corporate IdP (Azure AD, Okta)
IAS can federate with corporate identity providers via SAML 2.0 or OIDC, enabling single sign-on with company credentials.
Authorization — "What Can You Do?"
☁️
XSUAA (SAP BTP)
Used in cloud-based Fiori apps (CAP/RAP on BTP). Roles and scopes defined in xs-security.json. Tokens issued as JWTs.
🖥️
PFCG Roles (ABAP)
Used in on-premise S/4HANA. Authorization objects and profiles assigned to users via transaction PFCG.
Access Matrix Example
User RoleCan Open HR App?Can Approve Leaves?Can View All Employees?
HR Manager✅ Yes✅ Yes✅ Yes
Employee✅ Yes (own data)❌ No❌ No
Team Lead✅ Yes✅ Team only❌ No
Guest❌ No❌ No❌ No
💡
Remember: Authentication and Authorization are separate. Authentication proves identity; Authorization controls access. In SAP, IAS handles authentication while XSUAA (BTP) or PFCG (ABAP) handles authorization.
Q: What is the difference between IAS and XSUAA?
✓ IAS (Identity Authentication Service) handles authentication — it verifies who the user is via login, MFA, and SSO. XSUAA handles authorization on SAP BTP — it issues OAuth tokens with scopes/roles that determine what the authenticated user can access. PFCG plays the XSUAA role in on-premise ABAP systems.
Layer 04 · Frontend

SAPUI5 Application

The client-side JavaScript application that runs in the browser, renders the UI, and orchestrates all communication with backend services.

Standard Project Structure
webapp/ │ ├── Component.js ← App entry point, lifecycle hooks ├── manifest.json ← App descriptor: routing, models, services │ ├── view/ ← XML views (the UI templates) │ ├── App.view.xml │ └── Overview.view.xml │ ├── controller/ ← Business logic, event handlers │ ├── App.controller.js │ └── Overview.controller.js │ ├── model/ ← JSON models, formatters │ ├── models.js │ └── formatter.js │ ├── i18n/ ← Localization (translatable texts) │ └── i18n.properties │ └── css/ ← Custom styles └── style.css
MVC Pattern
🖼️
Model (M)
Holds the data. JSONModel for client-side data, ODataModel V4 for backend data. Supports property, aggregation, and element binding.
📄
View (V)
XML view defines the UI structure declaratively. Uses SAPUI5 controls like sap.m.Table, sap.m.Page, sap.m.Button. Bound to model via binding syntax.
⚙️
Controller (C)
Handles events, lifecycle hooks (onInit), and user interactions. Extends sap.ui.core.mvc.Controller. Implemented via sap.ui.define.
Bootstrap Script (How SAPUI5 Loads)
<script id="sap-ui-bootstrap" src="resources/sap-ui-core.js" data-sap-ui-theme="sap_horizon" data-sap-ui-libs="sap.m" data-sap-ui-compatVersion="edge" data-sap-ui-async="true" data-sap-ui-onInit="module:sap/ui/core/ComponentSupport" ></script>
Q: What is the role of manifest.json in an SAPUI5 application?
✓ manifest.json is the application descriptor. It defines the app's metadata (sap.app), UI settings (sap.ui), and SAPUI5-specific configuration (sap.ui5) — including routing, models, content densities, and service endpoints. The FLP reads it to configure tiles and the framework reads it to bootstrap the app.

Q: Why doesn't the SAPUI5 frontend connect directly to SAP HANA?
✓ Direct DB access from a browser would be a severe security risk (credentials exposed, no auth control, no business logic enforcement). OData services provide a secure, standardized API layer that enforces authorization, applies business rules, and returns only the data the user is permitted to see — in a standardized JSON format.
Layer 06 · Logic

Business Logic Layer

Where SAP business rules are enforced — validations, calculations, workflows, and data operations — across three main implementation paradigms.

Three Paradigms
⚡ ABAP — Traditional On-Premise ▾
Stack: Fiori → SAP Gateway → ABAP Class → Business Logic → DB

ABAP (Advanced Business Application Programming) is SAP's proprietary language for S/4HANA on-premise. Business logic lives in ABAP classes, function modules, and BAPIs. OData services in ABAP are built using the SEGW transaction or Code-Based OData V4.
☁️ CAP — Cloud Application Programming Model ▾
Stack: Fiori → CAP Service (Node.js or Java) → service.js → SAP HANA

CAP is SAP's recommended framework for building cloud-native services on SAP BTP. You define a CDS (Core Data Services) schema, implement event handlers in Node.js or Java, and CAP auto-generates OData V4 endpoints. Excellent for new greenfield applications.
🏛️ RAP — ABAP RESTful Application Programming Model ▾
Stack: Fiori → RAP Behavior Definition → Business Objects → DB

RAP is the recommended approach for building transactional Fiori apps on S/4HANA Cloud. It uses ABAP CDS views + behavior definitions + behavior implementations to create enterprise-grade OData V4 services with built-in draft handling, validations, and actions.
Example Business Rule (Leave Validation)
// ABAP or CAP — pseudo-code logic on before CREATE LeaveRequest { requestedDays = leaveRequest.EndDate - leaveRequest.StartDate; availableBalance = getLeaveBalance(employeeID); if (requestedDays > availableBalance) { reject( message: "Insufficient leave balance" ); } if (leaveRequest.StartDate < today()) { reject( message: "Cannot request past dates" ); } // Passes validation → write to database }
Q: When would you choose CAP over ABAP for business logic?
✓ Choose CAP for new cloud-native apps on SAP BTP, especially when the team is comfortable with Node.js or Java and you need rapid development with auto-generated OData services. Choose ABAP/RAP when integrating with existing S/4HANA business logic, using standard SAP business objects, or building on S/4HANA Cloud where RAP is the strategic direction.

SAP HANA — Database Layer

The in-memory, column-store database that powers all SAP applications. It stores every piece of business data — from employees to purchase orders.

What SAP HANA Stores
👥
HR Data
Employees, leave balances, payroll, org structure, headcount, performance data.
🛒
Procurement
Purchase orders, vendors, goods receipts, invoices, contracts.
💰
Finance
GL accounts, cost centers, profit centers, sales orders, financial statements.
📦
Supply Chain
Materials, stock levels, warehouses, deliveries, transfer orders.
🎫
Customer Data
Customers, sales agreements, service orders, CRM data.
📈
Analytics
Real-time embedded analytics, aggregations, KPIs — processed in-memory at very high speed.

DB Access Flow
// How data reaches the browser from HANA Browser (SAPUI5 ODataModel) ↓ GET /Employees OData Service (CAP / RAP / ABAP) ↓ SELECT EmployeeID, Name FROM Employees SAP HANA (In-Memory Column Store) ↓ Result set (microseconds) Business Logic Layer ↓ Apply filters / auth checks OData Response: { "value": [...] } ↓ SAPUI5 Table renders data on screen ✓
Q: Why is SAP HANA faster than traditional relational databases for analytics?
✓ SAP HANA stores data in-memory (eliminating disk I/O) and uses a column-oriented store, which is ideal for aggregations — only the relevant columns are read, not entire rows. This enables real-time OLTP and OLAP in the same system without separate data warehouses.
End-to-End

Complete Request Flow

Trace a real user action — opening "My Leave Requests" — through every layer of the architecture from click to data display.

Interactive Flow (click any step)
👤 Employee opens Fiori Launchpad
↓
🔑 IAS — Authentication (Login)
↓
🔐 XSUAA / PFCG — Authorization Check
↓
🏠 FLP Renders Authorized Tiles
↓
🖱️ User Clicks "Leave Request" Tile
↓

↓
📡 GET /LeaveRequests (OData V4)
↓
⚙️ CAP / ABAP Service Handles Request
↓

↓
📊 JSON → SAPUI5 Table Displayed ✓
Step 1 — User Access: The employee navigates to the company's Fiori Launchpad URL in their browser (Chrome/Edge). The browser makes an HTTPS request to the SAP BTP or ABAP server hosting the FLP.
Step 2 — Authentication: IAS (Identity Authentication Service) intercepts the request. If the user has no valid session, they are redirected to the login screen. After entering credentials (or via SSO from Azure AD), IAS creates a session token.
Step 3 — Authorization: XSUAA (BTP) or PFCG (ABAP) checks what roles the authenticated user has. Only tiles and apps the user is authorized for are rendered on the launchpad.
Step 4 — FLP Renders: The Fiori Launchpad shell loads. It reads the user's role assignments and displays only authorized tiles. The employee sees their ESS (Employee Self Service) tiles.
Step 5 — Tile Click: The user clicks the "Leave Request" tile. The FLP reads the target application URL from the tile configuration and initiates navigation using the SAP Router or hash-based routing.
Step 6 — App Load: The browser downloads the SAPUI5 application files (JavaScript, XML views, CSS, i18n). Component.js initializes, manifest.json is read, the router starts, and the initial view (e.g., Overview) renders.
Step 7 — OData Request: The SAPUI5 ODataModel V4 automatically sends GET /LeaveRequests?$filter=EmployeeID eq '101' to the backend OData service. This is triggered by the list binding on the Table control.
Step 8 — Service Layer: The CAP or ABAP service receives the request. It checks authorization (XSUAA scope), applies business logic filters (employee can only see own records), and queries SAP HANA via SQL/CDS.
Step 9 — Database: SAP HANA executes the SELECT statement against the LeaveRequest table. The in-memory column store returns matching records in sub-millisecond time. The result set is passed back to the service layer.
Step 10 — Display: The OData service serializes the result as JSON. SAPUI5's ODataModel receives the response, updates the model data, and the bound Table control automatically re-renders with the employee's leave records. The round-trip is complete. ✓

Interview Answer (memorize this):

"A user logs in through the SAP Fiori Launchpad. IAS authenticates the user and XSUAA or PFCG authorizes their access. Self-Assessment Quiz
Q1: A user can open the app but cannot see the "Delete" button. Which layer is responsible?
✓ Layer 3 — Authorization (XSUAA or PFCG). The user is authenticated but lacks the authorization scope/role required to perform delete operations. The UI can hide the button dynamically using device/model binding based on the user's scope, or the backend can reject unauthorized DELETE requests.
Q2: Where in the architecture does "Leave Balance < Requested Days → Reject" logic live?
✓ Layer 6 — Business Logic (CAP, RAP, or ABAP). This validation runs server-side in the service layer before any database write occurs. It is NOT implemented in SAPUI5 (frontend validation alone is insufficient — it can be bypassed).
Q3: What format does data arrive in when SAPUI5 receives it from an OData V4 service?
✓ JSON. OData V4 exclusively uses JSON format (unlike V2 which supported XML). The response has a standard structure with a "value" array for entity sets. SAPUI5's ODataModel V4 parses this automatically and updates the model, triggering UI re-render via data binding.
