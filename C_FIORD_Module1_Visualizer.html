What is SAP Fiori?

Lesson 2

SAP Fiori is SAP's design language and UX approach — it covers Design, Principles, and Technology for consistent, role-based user experiences across all SAP products.

↑ The 3 Dimensions × 5 Principles framework

🏛️ 3 Dimensions

  • Design — visual guidelines, Fiori Design System
  • Principles — the 5 principles (below)
  • Technology — SAPUI5, OData, Fiori Elements

⭐ 5 Design Principles

  • Role-Based — tailored to the user's job
  • Adaptive — works on any device
  • Coherent — consistent look & feel
  • Simple — only what's needed
  • Delightful — pleasant to use
💡
Exam tip: The 5 principles acronym: RACSD — Role-based, Adaptive, Coherent, Simple, Delightful.

SAP S/4HANA — Evolution & Reimaginations

Lesson 11

↑ SAP R/2 (1979) → R/3 (1992) → ERP (2004) → S/4HANA (2015)

79

SAP R/2 — 1979

Mainframe-based ERP. First SAP product.

92

SAP R/3 — 1992

Client-server architecture. GUI-based. Revolutionary for its time.

04

SAP ERP (ECC) — 2004

SOA-based. Web Dynpro introduced. Foundation for modern SAP.

15

SAP S/4HANA — 2015

In-memory. SAP Fiori UX. S = Suite, 4 = 4th generation. HANA database mandatory.

↑ S/4HANA's 3 reimaginations

🎨 User Experience

SAP Fiori UX replaces all GUI transactions. Role-based, beautiful, consistent.

📊 Data Models

Simplified schema. In-memory storage on HANA. No aggregates. Reduced footprint.

⚡ System Configuration

Rapid activation replaces lengthy implementation projects. Task list-driven.

⚠️
Exam trap: "Data Models" is about schema simplification — not just changing the database. Don't confuse with "switching from Oracle to HANA".

ABAP Programming Models & Fiori Elements

Lesson 12

↑ Old: "Data to Code" (bring DB data to ABAP) → New: "Code to Data" (push logic to HANA)

4 ABAP Programming Models (Evolution)

① Code-Based (Classic) — Manual ABAP + SEGW ▼

Write ABAP functions/RFCs/BAPIs manually. Use SEGW to build OData service. Generated classes: DPC (Data Provider Class) + MPC (Model Provider Class). You fill DPC with custom ABAP logic.

② CDS + SADL — Auto-generated OData from CDS Views ▼

SADL (Service Adaptation Definition Language) auto-generates OData services from CDS Views — no manual DPC coding for reads. CDS Views deployed as HANA database views. Calculation Views can sit below for analytics.

③ CDS + BOPF / RAP — Transactional with Managed Runtime ▼

RAP (ABAP RESTful Application Programming) — ABAP Platform 7.54+. Adds Managed Runtime (handles lock/draft/activate automatically) and Draft Table in HANA for save-without-commit. RAP is the future of SAP Fiori development.

↑ ABAP Development Tools (ADT) in Eclipse — CDS Editor, Preview, Outline panels

Migrating Non-Typed Technical Catalogs

Lessons 24–25
📚
Context: Older SAP systems used "non-typed" technical catalogs — flat catalogs containing both tiles and target mappings without a clear structure. The new content model (since S/4HANA 2023) uses typed catalogs with clear separation between Standard Catalogs (TC) and Business Catalogs (BC).

Migration Path

🗂️ Old: Non-Typed TC

Technical catalog with tiles + target mappings mixed together. No clear Business/Standard split.

✅ New: Typed Catalog

Standard Catalog (TC) holds app descriptors. Business Catalog (BC) references tiles for users.

Migration Steps

1

Identify Non-Typed Catalogs

Use FLPAM to find catalogs that don't have a Standard or Replicable type assigned.

2

Assign Catalog Type

In FLPAM, select the catalog and assign the correct type (Standard Catalog or Replicable Catalog).

3

Create Business Catalogs

Copy from the now-typed technical catalog into a BC using FLPCM. References auto-update.

4

Assign BCs to Roles

In PFCG, replace old TC references with the new BC references in business roles.

⚠️
Important: Standard and Replicable catalog types must be enabled in FLPAM settings before you can assign them. Without this step, the type dropdown is unavailable.

🧠 Unit 5 Quiz

Q1: What is a "non-typed" technical catalog and why does it need migration?
✅ An older-style technical catalog without a Standard or Replicable type. Needs migration to fit the new content model (typed TC → BC reference structure).
Q2: What must you enable in FLPAM before assigning catalog types?
✅ Enable "Standard and Replicable Catalogs as Catalog Types" in FLPAM settings.

🎯 Unit 5 Key Takeaways

  • Non-typed TCs are legacy — need migration to typed (Standard/Replicable) TCs
  • Migration path: Identify → Assign Type (FLPAM) → Create BC references (FLPCM) → Assign to roles (PFCG)
  • Enable catalog types in FLPAM settings first

Runtime Adaptation (RTA) & App Variants

Lesson 33

↑ App variant workflow (since S/4HANA 2023): FLP → FLPAM → FLPCM → FLP adapt → Activate

👤
Key User Extensibility allows non-developers to adapt SAP Fiori UIs without coding. Prerequisites: role SAP_UI_FLEX_KEY_USER + original app assigned + RTA plugin assigned.

App Variant Creation Flow (S/4HANA 2023+)

1

FLP: Create App Variant

Start original app → User Menu → Adapt UI → More Actions → App Variants → Save As. Copy the App Variant ID.

2

FLPAM: Define App Descriptor

Copy original app descriptor to a Standard Catalog. Replace SAPUI5 Component ID with the App Variant ID.

3

FLPCM: Assign to Role

Reference the target mapping in a Business Catalog. Assign BC to role.

4

FLP: Adapt the Variant

Start the app variant → Adapt UI → Make changes (rename groups, add/remove fields).

5

FLP: Activate & Publish

Activate as new version → Publish to transport request (customizing = client-specific, workbench = cross-client).

🔑
Version note: Versions of app variants were introduced in S/4HANA 2023. Before this, key users directly adapted the original app and saved as a variant (no versions).

// Get FLP renderer to modify shell
rendererPromise.then(function(oRenderer) {
  oRenderer.setHeaderTitle("My Company");
});
⚠️
Plugins load asynchronously — never assume a specific instantiation order when multiple plugins are active. Every plugin must be defined in /UI2/FLP_CONF_DEF before it can be activated. See SAP Note 3085381 for the full guide.

🎯 Unit 7 Key Takeaways

  • UI Theme Designer: browser-based, transaction /UI5/THEME_DESIGNER
  • Default theme: SAP Morning Horizon (sap_horizon)
  • App variant flow (2023+): FLP → FLPAM → FLPCM → FLP adapt → Activate version
  • Key user role: SAP_UI_FLEX_KEY_USER
  • FLP Plugins defined in /UI2/FLP_CONF_DEF, type="plugin" in manifest.json
  • Plugins load asynchronously — no guaranteed order
